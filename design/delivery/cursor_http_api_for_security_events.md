## 1. 问题定义

我建议把问题先定义为：

> 为外部可信服务提供一个标准化安全事件提交通道，使 Security Center 能在事件产生后 X 秒内接收并持久化，并让安全运营人员能够在 Web 端按事件类型、来源、时间、严重级别检索事件并查看详情。

你需要先回答：

**关键问题：这个接口解决的是“采集安全事件”，还是“触发安全处置流程”？**

推荐答案：第一阶段只解决“采集与展示”，不承诺自动处置、告警编排、工单流转。否则范围会失控。

验收标准：

- 外部服务提交合法事件后，后台必须返回明确成功响应。
- Security Center 后台必须能查到该事件。

## 2. MECE 决策树

我按“业务闭环”拆解，维度是：谁提交、提交什么、系统如何承接、谁消费、如何验收。它们覆盖从事件产生到人工查看的完整链路，且互不重叠。

### 第一层：业务闭环

1. 外部接入方
2. 事件定义
3. 接收与持久化
4. Web 展示与运营消费
5. 治理与验收

### 第二层：外部接入方

外部接入方必须拆成三类，MECE：

1. 已知可信服务：公司内部或已注册系统
2. 半可信合作方：外部系统但有合作关系
3. 未知调用方：没有接入授权

批判性问题：你是否允许“任何外部服务”直接提交事件？

推荐答案：不允许。必须先注册接入方，每个接入方有唯一身份、密钥、允许提交的事件类型、速率限制和责任人。

验收测试：

- 控制点：使用已注册服务凭证提交事件。
  观测点：返回成功，事件中记录 source/service identity。
- 控制点：使用无效凭证提交事件。
  观测点：请求被拒绝，事件不进入业务事件表，但进入安全审计日志。
- 控制点：注册服务提交未授权事件类型。
  观测点：请求被拒绝，并能看到拒绝原因。

### 第三层：事件定义

事件定义必须拆成三类，MECE：

1. 事件身份：事件 ID、类型、版本、来源
2. 事件内容：字段、字段类型、必填规则、枚举值
3. 事件业务含义：严重级别、处置建议、展示名称、分类

批判性问题：你说“通过配置文件定义事件的 ID 和内容格式”，配置文件是给谁维护的？

推荐答案：第一阶段由 Security Center 团队维护，不开放给外部服务自助修改。外部服务只能按已发布 schema 提交事件。

进一步追问：

- 事件 ID 是“事件类型 ID”，还是“单次事件实例 ID”？
- 内容格式是否需要版本化？
- 当配置变更后，历史事件按旧格式展示，还是迁移到新格式？

推荐答案：

- `eventTypeId` 表示事件类型，例如 `LOGIN_BRUTE_FORCE`。
- `eventId` 表示单次事件实例，由调用方传入或服务端生成。
- schema 必须有版本，例如 `eventTypeId + version`。
- 历史事件保留原始 payload 和 schemaVersion，不做强制迁移。

验收测试：

- 控制点：提交已配置事件类型和合法字段。
  观测点：事件成功保存，详情页按配置展示字段。
- 控制点：提交未知事件类型。
  观测点：请求失败，返回明确错误码。
- 控制点：提交缺少必填字段的事件。
  观测点：请求失败，指出缺失字段。
- 控制点：提交旧版本 schema 的历史事件。
  观测点：详情页仍能按旧版本正确展示。

继续。现在进入第二个核心分支：**配置文件到底定义什么**。这是整个方案最容易失控的地方。

我先给结论：**第一版配置文件只能定义“事件契约”，不能定义“业务逻辑”。**  
也就是说，它可以定义事件类型、字段、校验规则、展示规则，但不要定义告警条件、处置动作、复杂表达式或动态计算逻辑。

## 1. 配置文件的业务边界

配置文件的职责可以按 MECE 拆成三类：

1. **接入约束**：谁能提交什么事件
2. **数据契约**：事件必须长什么样
3. **展示元数据**：Web 端如何展示这个事件

这三个维度互不重叠：

- 接入约束回答“是否允许提交”
- 数据契约回答“提交内容是否合法”
- 展示元数据回答“安全运营如何理解”

它们合起来覆盖了“接收前校验”和“接收后展示”的完整需求。

## 2. 第一层：接入约束

### 第二层拆解

接入约束必须包含：

1. 来源系统清单
2. 来源系统允许提交的事件类型
3. 事件类型当前可用状态

### 第三层追问

关键问题：是否允许一个内部系统提交所有事件类型？

我的推荐答案：**不允许。**  
每个 `sourceSystem` 必须显式配置允许提交的 `eventTypeId`。否则一旦某个内部系统误用接口，会污染所有安全事件域。

建议配置含义：

- `sourceSystem`：来源系统唯一标识
- `allowedEventTypes`：该系统允许提交的事件类型列表
- `enabled`：是否允许继续接入

验收测试：

- 控制点：`iam-service` 提交被允许的 `LOGIN_BRUTE_FORCE`。
  观测点：提交成功。

- 控制点：`iam-service` 提交未被允许的 `DATABASE_LEAK`。
  观测点：提交失败，事件不入库。

- 控制点：已禁用来源系统提交事件。
  观测点：提交失败，并返回来源系统不可用。

## 3. 第二层：数据契约

### 第二层拆解

数据契约必须包含：

1. 事件基础字段
2. payload 字段定义
3. schema 版本规则

### 第三层追问

关键问题：payload 是否允许完全自由 JSON？

我的推荐答案：**不允许完全自由。**  
第一版必须做“有限 schema 校验”：字段名、字段类型、必填、枚举、最大长度。否则 Web 展示会变成 JSON 垃圾桶，安全运营无法稳定消费。

建议支持的字段类型第一版只保留：

- `string`
- `number`
- `boolean`
- `datetime`
- `enum`
- `object`
- `array`

不建议第一版支持：

- 条件必填
- 跨字段校验
- 正则表达式集合
- 动态计算字段
- 嵌套过深的复杂对象

批判性问题：为什么限制这么死？

因为你的第一版目标是“稳定采集与展示”，不是做一个通用事件规则引擎。越早支持复杂 schema，越早把后端校验、前端展示、配置维护全部复杂化。

验收测试：

- 控制点：payload 缺少必填字段。
  观测点：提交失败，返回字段名。

- 控制点：字段类型错误，例如 `riskScore` 应为 number 但传 string。
  观测点：提交失败，返回类型错误。

- 控制点：枚举字段传入非法值。
  观测点：提交失败，返回允许值范围。

- 控制点：payload 包含配置外字段。
  观测点：我的建议是第一版允许保存但标记为 `extraFields`，详情页展示在“未定义字段”区域。

这里有一个必须拍板的问题：

**配置外字段是拒绝，还是允许保存？**

我的推荐：**允许保存，但不参与结构化展示。**  
原因是安全事件可能有临时上下文，完全拒绝会降低接入成功率；但必须明确标记，避免运营误以为它是正式字段。

## 4. 第三层：展示元数据

### 第二层拆解

展示元数据必须包含：

1. 列表展示字段
2. 详情展示字段
3. 字段的人类可读解释

### 第三层追问

关键问题：Web 列表的摘要来自调用方，还是后台配置生成？

我的推荐答案：**调用方提交 `summary`，配置文件定义该事件类型的展示名称和字段标签。**

不要第一版做模板渲染，例如：

`用户 {username} 在 {ip} 登录失败 {count} 次`

这看起来方便，但会引入模板语法、字段缺失处理、多语言、转义、安全显示等问题。

第一版建议：

- 列表直接展示 `summary`
- 详情页展示字段 label
- 事件类型显示配置中的 `displayName`
- 严重级别显示提交值或配置默认值

验收测试：

## 5. 幂等与重复事件

### 第二层拆解

1. 同一来源、同一 `eventId`、内容完全相同
2. 同一来源、同一 `eventId`、内容不同
3. 不同来源、同一 `eventId`

### 第三层规则

推荐规则：

- 同一来源、同一 `eventId`、内容完全相同：返回成功，并标记为重复提交，不新增业务事件。

- 不同来源、同一 `eventId`：允许保存为不同事件。

- 控制点：不同 `sourceSystem` 使用相同 `eventId`。

## 5. 当前最终业务规格

到这里，第一版规格已经闭合：

- 内部系统提交事件。
- 不做鉴权。
- 但 `sourceSystem` 必须在配置清单中。
- 每个来源系统只能提交被允许的事件类型。
- `eventTypeId + schemaVersion` 决定 payload schema。
- 配置外字段允许保存，但标记为未定义字段。
- `summary` 由调用方提交。
- 成功返回前必须完成持久化。
- 非法事件进入失败接收记录，不进入业务列表。
- 幂等键为 `sourceSystem + eventId`。
- 同一幂等键内容不同拒绝。
- 不考虑配置变化。
- 暂不设置默认保存期限。
- Web 面向安全运营，只做列表和详情查看。

## 总体验收目标

第一版上线后，必须证明四件事：

1. 内部系统能提交安全事件。
2. 后台能按配置判断事件是否合法。
3. 合法事件能被持久化并展示。
4. 非法事件不会污染安全运营列表，但可被追溯。

这里按业务链路做 MECE 拆解：

1. API 提交验收
2. 配置校验验收
3. 持久化验收
4. Web 展示验收
5. 异常与审计验收

这五类覆盖从外部系统提交到安全运营查看的完整闭环，且互不重叠。

## 1. API 提交验收

### 用例 1：合法事件提交成功

控制点：内部系统提交包含 `sourceSystem`、`eventId`、`eventTypeId`、`schemaVersion`、`severity`、`occurredAt`、`summary`、`payload` 的合法事件。

观测点：API 返回成功；事件被保存；Web 列表可见。

### 用例 2：缺少必填基础字段

控制点：提交缺少 `sourceSystem` 或 `eventTypeId` 的事件。

观测点：API 返回失败；失败原因指明缺失字段；事件不进入业务事件列表。

### 用例 3：调用方不传 `eventId`

控制点：合法事件缺少 `eventId`。

观测点：后台生成事件实例 ID；事件保存成功；详情页展示服务端生成的 `eventId`。

### 用例 4：调用方提交 `receivedAt`

控制点：请求体中包含 `receivedAt`。

观测点：后台忽略调用方传入值，使用服务端接收时间。

## 2. 配置校验验收

### 用例 5：来源系统在清单中

控制点：`sourceSystem` 是配置允许的内部系统。

观测点：继续进入事件类型与 payload 校验。

### 用例 6：来源系统不在清单中

控制点：提交未知 `sourceSystem`。

观测点：API 拒绝；业务事件列表不可见；失败接收记录可查。

### 用例 7：来源系统提交未授权事件类型

控制点：`sourceSystem` 存在，但提交了未配置在 `allowedEventTypes` 中的 `eventTypeId`。

观测点：API 拒绝；失败原因是来源系统无权提交该事件类型。

### 用例 8：未知事件类型

控制点：提交配置中不存在的 `eventTypeId`。

观测点：API 拒绝；失败原因明确为未知事件类型。

### 用例 9：未知 schema 版本

控制点：`eventTypeId` 存在，但 `schemaVersion` 不存在。

观测点：API 拒绝；事件不入业务列表。

### 用例 10：payload 缺少必填字段

控制点：payload 缺少配置中声明为必填的字段。

观测点：API 拒绝；返回缺失字段名。

### 用例 11：payload 字段类型错误

控制点：配置要求 `riskScore` 为 `number`，调用方传入字符串。

观测点：API 拒绝；返回字段类型错误。

### 用例 12：payload 枚举值非法

控制点：配置要求 `action` 只能是 `login/logout/block`，调用方传入 `delete`。

观测点：API 拒绝；返回允许值范围。

### 用例 13：payload 包含配置外字段

控制点：payload 中包含 schema 未定义字段。

观测点：事件允许保存；未定义字段不进入列表主列；详情页在“未定义字段”区域展示。

## 3. 持久化验收

### 用例 14：成功返回前完成持久化

控制点：提交合法事件后立即查询列表。

观测点：事件可被查询到；不存在“API 成功但事件还未保存”的状态。

### 用例 15：数据库写入失败

控制点：模拟数据库不可写。

观测点：API 返回失败；不能返回成功。

### 用例 16：重复提交完全相同事件

控制点：同一 `sourceSystem + eventId` 提交完全相同内容。

观测点：API 返回成功；业务事件列表只出现一条；可识别为重复提交。

### 用例 17：重复 eventId 但 schema 不同

控制点：同一 `sourceSystem + eventId` 提交不同 `eventTypeId` 或 `schemaVersion`。

观测点：第二次请求被拒绝；记录幂等冲突。

### 用例 18：重复 eventId 且 schema 相同但字段值不同

控制点：同一 `sourceSystem + eventId`，payload 字段值不同。

观测点：第二次请求被拒绝；提示该事件 ID 已绑定不同事件事实。

### 用例 19：不同来源使用相同 eventId

控制点：不同 `sourceSystem` 使用相同 `eventId`。

观测点：两条事件均可保存；互不冲突。

## 4. Web 列表验收

### 用例 20：默认列表展示

控制点：安全运营打开事件列表页。

观测点：按 `receivedAt` 倒序展示事件。

### 用例 21：列表展示核心字段

控制点：查看任意事件列表项。

观测点：能看到接收时间、发生时间、来源系统、事件类型展示名、严重级别、摘要、事件 ID。

### 用例 22：按来源系统筛选

控制点：选择某个 `sourceSystem`。

观测点：列表只展示该来源系统提交的事件。

### 用例 23：按事件类型筛选

控制点：选择某个事件类型。

观测点：列表只展示该类型事件。

### 用例 24：按严重级别筛选

控制点：筛选 `high` 或 `critical`。

观测点：列表只展示对应严重级别事件。

### 用例 25：按时间范围筛选

控制点：选择发生时间或接收时间范围。

观测点：列表只展示范围内事件。

## 5. Web 详情验收

### 用例 26：进入详情页

控制点：安全运营点击列表中的某条事件。

观测点：进入该事件详情页，且 URL 可重新打开定位同一事件。

### 用例 27：展示基础事实

控制点：查看事件详情。

观测点：显示 `eventId`、`sourceSystem`、`eventTypeId`、事件类型展示名、`schemaVersion`、`severity`、`summary`、`occurredAt`、`receivedAt`。

### 用例 28：展示结构化 payload

控制点：payload 字段在配置中定义了 label。

观测点：详情页使用 label 展示字段，而不是只显示原始字段名。

### 用例 29：展示未定义字段

控制点：payload 包含配置外字段。

观测点：详情页单独展示为“未定义字段”。

### 用例 30：展示原始 payload

控制点：打开详情页查看原始内容。

观测点：可以看到原始提交 payload；内容较大时页面不应卡死。

## 6. 失败记录验收

### 用例 31：非法事件不进入业务列表

控制点：提交未知来源、未知事件类型或 schema 错误事件。

观测点：Web 业务事件列表不可见。

### 用例 32：失败记录可追溯

控制点：提交非法事件后查看失败接收记录。

观测点：能看到接收时间、来源字段、事件类型字段、失败原因和请求摘要。

### 用例 33：超大非法 payload

控制点：提交超大非法 payload。

观测点：系统拒绝或安全记录失败摘要；不会造成存储或页面不可用。
