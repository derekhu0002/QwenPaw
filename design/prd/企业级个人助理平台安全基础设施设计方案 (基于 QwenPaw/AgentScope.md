# 企业级个人助理平台安全基础设施设计方案 (基于 QwenPaw/AgentScope)

## 1. 项目概述
本项目旨在基于开源项目 QwenPaw (AgentScope 框架) 构建公司自研的个人助理平台。安全基础设施是该平台的底层支柱，核心任务是构建一套**“不信任 Agent 逻辑、不信任插件工具、不信任框架源码”**的零信任安全防御体系。

---

## 2. 核心设计架构：中枢拦截器管道 (Interceptor Pipeline)
为实现低侵入性且强掌控力的安全管理，摒弃单纯的“探针”或“硬编码修改”，采用**插件化拦截器管道**模式。

*   **设计决策：** 在 `AgentBase.reply` (语义层) 和 `ServiceFactory.execute` (执行层) 植入统一的 **Gatekeeper（门控模块）**。
*   **决策依据：** 
    *   **低侵入：** 仅需在基类关键路径修改几行代码，不影响 AgentScope 核心逻辑演进。
    *   **强顺序：** 严格保证“审计预记录 -> 权限预检 -> 业务执行 -> 响应脱敏 -> 审计终记录”的时序逻辑。
    *   **异常拦截：** 允许安全模块在不崩溃主程序的前提下，捕获异常并将其转换为礼貌的 Agent 回复。

---

## 3. 权限管理模块 (Permission Management)

### 3.1 身份核实与集成
*   **决策结论：** 禁用自建账号，全面对接公司标准 **OIDC/SAML (如 Keycloak)**。引入 **Token Exchange 机制**。
*   **决策依据：** 解决 Agent 长效会话与 SSO Token 短期过期的冲突。后端将用户 Token 交换为受限的系统级 Service Account Token，支持异步长耗时任务且权限可追溯。

### 3.2 双重鉴权架构 (Double Authentication)
*   **决策结论：** 实施“应用层 (RBAC/ABAC) + 网关层 (Zero Trust)”的双重校验。
*   **决策依据：** 防止 Agent 绕过应用逻辑直接调用后端 API。应用层校验“用户是否有权启动该工具”，网关层校验“请求是否来自合法的 Agent 运行环境”。

### 3.3 工作负载身份 (Workload Identity)
*   **决策结论：** 为每一个 Agent 会话发放包含 `User_ID`, `Agent_ID`, `Session_ID` 的**会话级受限 Token**。
*   **决策依据：** 即使发生 Prompt 注入攻击，攻击者也无法跨会话访问数据。

### 3.4 最小特权原则 (Least Privilege)
*   **决策结论：** 工具调用权限取“用户权限”与“Agent 授权清单”的**交集**。
*   **决策依据：** 限制 Agent 因幻觉或被恶意诱导而触发用户本人虽有权但 Agent 不应具备的操作（如删除数据库）。

---

## 4. 日志审计模块 (Log Auditing)

### 4.1 全链路追溯 (V1 阶段)
*   **决策结论：** 记录包含 `TraceID` 的结构化日志，涵盖：用户输入、CoT 思考过程、工具调用参数、工具返回结果、最终输出。
*   **决策依据：** 审计必须能复原 Agent 的决策黑盒，仅记录输入输出无法定位是 Agent 误操作还是指令注入。

### 4.2 实时脱敏与合规 (V2 阶段)
*   **决策结论：** 在日志落盘前，通过 **DLP (数据泄露防护) 网关**，利用正则和 NER 算法对 PII（个人隐私信息）进行掩码处理。
*   **决策依据：** 防止审计系统本身成为敏感数据泄露源，解决“谁来审计审计员”的问题。

### 4.3 异步外刷与不可篡改
*   **决策结论：** 采用生产者-消费者模型，通过独立 Sidecar 进程将日志推送到公司统一的只读日志中心 (如 Kafka -> ELK)。
*   **决策依据：** 确保性能不被审计逻辑拖累；确保即便应用服务器被攻破，历史审计日志不可被删除或篡改。

### 4.4 异常行为监控 (V3 阶段)
*   **决策结论：** 基于审计流建立 **UEBA (用户行为分析)**，对高频报错、高频工具调用等异常模式实施自动熔断。

---

## 5. 会话加密模块 (Session Encryption)

### 5.1 信封加密架构 (Envelope Encryption)
*   **决策结论：** 采用“一会话一密钥”。使用对称密钥 (DEK) 加密对话，DEK 由公司 KMS 中的主密钥 (KEK) 加密。
*   **决策依据：** 数据库泄露不代表数据泄露；支持用户级数据的物理级彻底销毁（仅需删除 KMS 中的 KEK）。

### 5.2 搜索与加密的平衡
*   **决策结论：** 采用 **“语义索引分层”** 方案。
    *   **明文流：** 转为不可逆向量 (Vector) 存入向量数据库，用于 RAG 检索。
    *   **加密流：** 原始文本加密存入关系库，仅在对话加载时解密。
*   **决策依据：** 满足 AI Agent 必须的语义检索需求，同时保证原始文本的存储安全。

### 5.3 内存安全
*   **决策结论：** 仅在调用 LLM 的瞬时解密；禁用生产环境 Core Dump；在会话结束时显式清理内存上下文。

## 6. 应急响应与自毁机制 (The "Kill Switch")

### 6.1 三位一体自毁设计
当平台遭遇 0-Day 漏洞或严重安全威胁时，启动以下机制：
1.  **凭证层：** 轮转全局安全盐值，立即使全平台所有已发放 Token 失效。
2.  **网关层：** 动态下发 ACL 规则，阻断所有 Agent 平台指向业务系统的 API 请求。
3.  **逻辑层：** 拦截器管道强制进入“只读模式”，仅允许纯文本交互，切断所有工具执行能力。

### 6.2 决策依据
确保在应用层代码被攻破（RCE）的情况下，依然能通过基础设施（网关/身份中心）层级强制锁死风险，实现“Assume Breach”的防御理念。

## 8. 实现设计补充：本机运行 + 外部 MCP/API 调用场景

### 8.1 需求与安全边界

本阶段的实现目标不是直接建设外部多租户 SaaS 平台，而是在 QwenPaw 本机运行模式下，控制个人助理对云侧 MCP 工具和外部 API 的访问风险。安全边界定义如下：

1.  **本机身份可信：** 本机控制台和本机后端必须能识别当前操作者，并支持登录、登出、Token 失效、Token 吊销和高危配置保护。
2.  **外部动作最小授权：** Agent 调用 MCP 工具、插件工具或外部 API 前，必须经过统一策略判断。未知服务、未知工具、未知动作默认拒绝。
3.  **工具执行可中断：** 高风险 MCP/API/工具调用必须能被拦截、审批、拒绝、超时终止或切换到只读模式。
4.  **审计可追责：** 权限判断、审批、工具调用、外部 API 调用、会话创建/删除、安全配置变更必须产生结构化审计事件。
5.  **本地审计防篡改感知：** 本地审计文件只能做到篡改可检测，不能承诺绝对不可删除。真正不可抵赖必须依赖公司日志中心、SIEM、WORM 存储、远端 append-only 存储或签名时间戳服务。
6.  **敏感数据加密：** 配置密钥、外部服务凭证、会话原文、工具入参和工具结果在落盘前必须经过加密或脱敏策略处理。
7.  **前端不作为最终授权点：** 控制台只展示权限状态和审批状态，最终授权必须由后端策略引擎执行。

本阶段明确不覆盖以下范围：外部多租户隔离、跨组织租户计费、完整企业 IAM 替换、全量 UEBA 平台建设。上述能力可作为后续演进方向。

### 8.2 当前系统实现现状和 GAP

### 8.3 方案计划

#### 8.3.1 总体实现路径

第一期采用“本地策略执行 + 本地防篡改感知审计 + 可选远端审计保全”的路径。低风险动作可先本地落盘再异步外发；高风险 MCP/API 调用必须支持远端审计写入成功后再执行。

执行顺序如下：

1.  **安全上下文标准化**
    *   定义 `SecurityContext`，统一携带 `trace_id`、`user_id`、`session_id`、`root_session_id`、`agent_id`、`tool_call_id`、`channel`、`mcp_server`、`resource_scope`。
    *   在请求入口、Agent 执行入口、工具执行入口和审批入口传递同一上下文。

2.  **权限策略层**
    *   新增本地策略引擎，输入为主体、Agent、工具/服务、动作、资源范围，输出为 `allow`、`deny`、`require_approval`、`require_remote_audit`。
    *   策略默认拒绝未知 MCP 服务、未知工具、未知动作和未知资源范围。
    *   工具调用权限取“用户权限”和“Agent 授权清单”的交集。
    *   复用现有 Tool Guard 风险检测结果，将风险等级纳入策略决策。

3.  **审计事件层**
    *   定义 `AuditEvent`：包含事件 ID、事件类型、时间、主体、会话、Agent、资源、动作、策略决策、审批结果、执行结果、脱敏状态、前序 hash、当前 hash。
    *   本地以 append-only JSONL 或 SQLite WAL 模式写入，并维护 hash chain。
    *   定期生成 checkpoint，记录最后连续事件 ID、checkpoint hash、签名和远端外发状态。
    *   明确本地审计是 tamper-evident，不是 tamper-proof。删除本地文件只能被后续启动校验、远端缺口或 checkpoint 缺失发现，不能被纯本地机制绝对阻止。
    *   对高风险事件支持同步外发公司日志中心；外发失败时拒绝执行或进入只读模式。

4.  **会话加密层**
    *   先盘点所有会话、对话、工具调用参数、工具结果的落盘路径。
    *   引入每会话 DEK；DEK 由本地 KEK 或公司 KMS KEK 包裹。
    *   第一阶段本机可复用 OS keyring 或现有 secret store 主密钥机制，接口预留 KMS 适配。
    *   原始会话内容加密保存；审计日志只保存脱敏摘要、hash、资源标识和必要元数据。

5.  **控制台与运维面**
    *   控制台展示权限策略结果、待审批请求、审批历史、审计事件查询和外发状态。
    *   桌面端优先使用系统安全存储保存 Token；浏览器端至少支持短期 Token、登出清理和 401 强制失效。
    *   所有安全配置变更必须走后端 API，并产生审计事件。

6.  **应急响应**
    *   增加全局只读模式：允许普通问答，禁止外部 MCP/API 写操作和高风险工具。
    *   增加全局 Token 吊销：复用现有 `revoke_all_tokens` 思路，统一触发会话失效和工具审批清空。
    *   增加外部服务阻断清单：高危状态下默认拒绝所有外部服务调用，仅保留显式允许的只读服务。

### 8.4 端到端验收规格

以下验收规格用于驱动后续实现和测试。每个规格必须同时定义控制点和观测点，避免只测 UI 或只测函数。

| ID | 需求 | 控制点 | 观测点 | 验收标准 |
| --- | --- | --- | --- | --- |
| SEC-E2E-001 | 未知 MCP 服务默认拒绝 | 在本机助理会话中触发一个未登记 MCP server 的工具调用。 | 后端策略引擎返回 `deny`，工具未执行，控制台显示拒绝原因，审计事件包含 `policy_decision=deny` 和 MCP server 标识。 | 未知服务不会发起网络调用；本地审计出现一条拒绝事件；若远端审计开启，远端也收到同一 `trace_id` 事件。 |
| SEC-E2E-002 | 高风险外部写操作必须先审计后执行 | 配置一个外部 API 写操作为高风险动作，并在远端审计 sink 可用时触发调用。 | 审计事件先写入本地 hash chain，再写入远端 sink，随后工具/API 才执行。 | 执行结果时间晚于远端审计确认时间；审计事件包含动作、资源范围、Agent、用户和执行结果。 |
| SEC-E2E-003 | 高风险外部写操作在远端审计失败时拒绝 | 关闭或模拟远端审计 sink 故障后触发同一高风险写操作。 | 策略结果为 `require_remote_audit`，远端写入失败，执行层返回拒绝或进入只读模式。 | 外部 API 没有被调用；本地审计记录外发失败和拒绝原因；控制台提示无法满足审计前置条件。 |
| SEC-E2E-004 | 工具调用审批闭环 | 触发一个 Tool Guard 判定为中高风险的工具调用，并在控制台或审批命令中批准。 | 审批请求进入 pending 状态；批准后工具执行；审批创建、批准和执行结果均写入审计事件。 | 同一 `trace_id` 能串联 pending、approved、executed 三类事件；审批记录重启后仍可查询。 |
| SEC-E2E-005 | 审批拒绝阻断工具执行 | 触发同一高风险工具调用并选择拒绝。 | 后端返回拒绝结果，工具不执行，审计记录审批拒绝。 | 无工具副作用；审计事件包含拒绝人、拒绝时间、拒绝原因和工具标识。 |
| SEC-E2E-006 | 本地审计篡改可检测 | 生成连续审计事件后，手动修改或删除中间一条本地审计记录，再重启后端或运行审计校验命令。 | 审计校验发现 hash chain 断裂、checkpoint 不匹配或事件 ID 缺口。 | 系统报告 tamper detected；控制台展示审计完整性异常；后续高风险外部调用被拒绝或要求重新建立可信状态。 |
| SEC-E2E-007 | 安全配置变更可追责 | 在控制台修改 MCP allowlist、工具风险等级或只读模式开关。 | 后端保存配置并写入审计事件，事件包含修改前值、修改后值、操作者和 `trace_id`。 | 配置生效；审计查询能按操作者、配置项和时间范围检索到该变更。 |
| SEC-E2E-008 | Token 吊销即时生效 | 登录后取得 Token，调用全局吊销或单 Token 吊销，再使用旧 Token 访问受保护 API。 | 认证中间件拒绝旧 Token；控制台收到 401 后清理本地 Token 并跳转登录页。 | 旧 Token 无法访问受保护接口；审计事件记录吊销和后续拒绝访问。 |
| SEC-E2E-009 | 会话原文落盘加密 | 创建一段包含敏感文本的会话并触发持久化。 | 直接读取会话落盘文件或数据库记录，只能看到密文、加密元数据或脱敏摘要。 | 明文敏感文本不出现在会话存储和审计存储中；通过合法会话加载路径可正常解密恢复。 |
| SEC-E2E-010 | 会话密钥隔离 | 创建两个不同会话，分别写入不同敏感内容，尝试用会话 A 的 DEK 解密会话 B 数据。 | 解密失败并产生安全异常审计事件。 | 不同会话密钥不能互相解密；异常不会泄露明文。 |
| SEC-E2E-011 | Prompt 注入诱导越权被拦截 | 在用户输入中诱导 Agent 调用未授权外部工具或扩大资源范围。 | 策略引擎基于实际工具、动作和资源范围判断，而不是信任模型输出文本。 | Agent 输出无法绕过策略；未授权工具未执行；审计事件记录 Prompt 来源、策略拒绝和风险标签。 |
| SEC-E2E-012 | 只读模式阻断高风险工具 | 启用全局只读模式后触发外部写 API、文件写入或命令执行。 | 策略引擎返回只读模式拒绝，允许普通问答继续运行。 | 高风险工具无副作用；普通对话仍可响应；审计事件记录只读模式拦截。 |
| SEC-E2E-013 | 控制台不承担最终授权 | 在浏览器开发者工具中绕过前端 UI，直接调用后端高风险工具 API。 | 后端仍执行认证、策略判断、审计前置和审批检查。 | 前端绕过不能导致未授权执行；后端返回拒绝或待审批；审计记录绕过式请求。 |
| SEC-E2E-014 | 审计外发断点续传 | 断开远端审计 sink 后产生低风险事件，再恢复远端 sink。 | 本地审计 exporter 从最后成功事件 ID 继续外发。 | 远端最终收到完整连续事件；无重复副作用；本地记录外发失败和恢复。 |
| SEC-E2E-015 | DLP 脱敏优先于审计落盘 | 触发包含手机号、邮箱、Token 或 API Key 的 Prompt 和工具参数。 | 审计写入前执行 DLP 脱敏，原始值只进入加密会话存储或不落盘。 | 审计事件中敏感字段被掩码或 hash；会话加密存储中无可直接检索的明文敏感值。 |

### 8.5 分阶段交付建议

| 阶段 | 交付物 | 验收重点 |
| --- | --- | --- |
| P0：安全上下文与审计骨架 | `SecurityContext`、`AuditEvent`、本地审计写入、hash chain、基础查询。 | SEC-E2E-006、SEC-E2E-007 的最小闭环。 |
| P1：外部 MCP/API 策略控制 | 策略引擎、MCP/API allowlist、deny-by-default、Tool Guard 决策融合。 | SEC-E2E-001、SEC-E2E-011、SEC-E2E-013。 |
| P2：高风险调用审计前置 | 远端审计 sink 接口、高风险动作远端确认后执行、外发失败拒绝。 | SEC-E2E-002、SEC-E2E-003、SEC-E2E-014。 |
| P3：审批持久化与控制台闭环 | 审批记录持久化、审批历史、控制台权限与审计视图。 | SEC-E2E-004、SEC-E2E-005、SEC-E2E-007。 |
| P4：会话加密与 DLP | 每会话 DEK、会话原文加密、审计脱敏、密钥隔离。 | SEC-E2E-009、SEC-E2E-010、SEC-E2E-015。 |
| P5：应急响应 | 全局只读模式、Token 全量吊销、外部服务阻断清单。 | SEC-E2E-008、SEC-E2E-012。 |
