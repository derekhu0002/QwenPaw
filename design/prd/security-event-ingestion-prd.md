# Security Center 安全事件接入 PRD

## 1. 背景

Security Center 需要接收公司内部系统产生的安全事件，并让安全运营人员能够在 Web 端查看事件列表和详情。当前目标不是建设完整的告警、工单或处置平台，而是先建立稳定的安全事件采集、校验、持久化和展示能力。

本 PRD 聚焦 V1 阶段业务需求，明确接口接入范围、事件配置规则、持久化语义、Web 展示要求和验收标准。

## 2. 目标

V1 目标是：

1. 公司内部系统可以通过 HTTP API 提交安全事件。
2. 后台可以通过配置文件定义允许的来源系统、事件类型、schema 版本和 payload 字段格式。
3. 后台收到合法事件后，必须先持久化保存，再返回成功。
4. Web 端为安全运营人员提供事件列表和事件详情查看能力。
5. 非法事件不能进入业务事件列表，但必须保留失败接收记录用于追溯。

## 3. 非目标

V1 明确不做：

1. 接入方身份管理。
2. 请求鉴权。
3. 第三方外部系统接入。
4. 自动告警。
5. 工单流转。
6. 通知推送。
7. 自动处置。
8. 事件状态流转。
9. 指派负责人。
10. 评论。
11. 导出报表。
12. 统计大屏。
13. Web 上自助编辑事件配置。
14. 配置变化后的历史 schema 展示策略。
15. 合法事件和失败记录的默认保存期限。

## 4. 用户与场景

### 4.1 主要用户

安全运营人员。

### 4.2 接入方

公司内部系统。

### 4.3 核心场景

1. 内部系统产生安全事件后，通过 HTTP API 提交给 Security Center。
2. Security Center 根据配置校验来源系统、事件类型、schema 版本和 payload 字段。
3. 合法事件被持久化保存。
4. 安全运营人员在 Web 列表页查看事件，并进入详情页查看完整内容。
5. 非法提交被拒绝，并进入失败接收记录。

## 5. 业务规则

### 5.1 接入范围规则

1. V1 仅允许公司内部系统提交事件。
2. V1 不做请求鉴权。
3. 所有事件必须声明 `sourceSystem`。
4. `sourceSystem` 必须存在于配置清单中。
5. 每个 `sourceSystem` 只能提交配置允许的 `eventTypeId`。
6. 未知来源系统、禁用来源系统、未授权事件类型均必须被拒绝。

### 5.2 事件实例规则

1. `eventId` 表示单次事件实例 ID。
2. 调用方可以提交 `eventId`。
3. 如果调用方不提交 `eventId`，后台生成事件实例 ID。
4. `eventId` 不是事件系列 ID，也不是聚合 ID。
5. 如果调用方需要提交新的事件事实，必须使用新的 `eventId`。

### 5.3 事件类型规则

1. `eventTypeId` 表示事件类型 ID。
2. `eventTypeId` 必须在配置文件中定义。
3. `schemaVersion` 表示事件格式版本。
4. `schemaVersion` 必须属于对应 `eventTypeId`。
5. `eventTypeId + schemaVersion` 共同决定 payload 校验规则。

### 5.4 payload 规则

1. payload 必须按配置文件进行字段校验。
2. V1 支持字段名、字段类型、必填、枚举值、最大长度校验。
3. payload 缺少必填字段时，请求必须被拒绝。
4. payload 字段类型错误时，请求必须被拒绝。
5. payload 枚举值非法时，请求必须被拒绝。
6. payload 出现配置外字段时，V1 允许保存，但必须标记为未定义字段。
7. 未定义字段不进入列表主列，只能在详情页的未定义字段区域展示。

### 5.5 summary 规则

1. `summary` 由调用方提交。
2. 后台只校验 `summary` 的必填和长度。
3. V1 不做后台模板生成摘要。

### 5.6 时间规则

1. `occurredAt` 表示事件发生时间，由调用方提交。
2. `receivedAt` 表示后台接收时间，由后台生成。
3. 如果调用方提交 `receivedAt`，后台必须忽略调用方传入值。

### 5.7 幂等规则

幂等键为 `sourceSystem + eventId`。

1. 同一来源、同一 `eventId`、规范化后内容相同：返回成功，不新增业务事件。
2. 同一来源、同一 `eventId`，但 `eventTypeId` 或 `schemaVersion` 不同：拒绝，记录幂等冲突。
3. 同一来源、同一 `eventId`，schema 相同但字段值不同：拒绝，记录幂等冲突。
4. 不同来源使用相同 `eventId`：允许保存为不同事件。

核心原则：同一个事件实例 ID 不能指向两个不同事件事实。

### 5.8 持久化规则

1. 合法事件必须持久化成功后，接口才能返回成功。
2. 数据库写入失败时，接口必须返回失败。
3. 不允许先返回成功再异步保存。
4. 合法事件进入业务事件列表。
5. 非法事件不进入业务事件列表。
6. 非法事件必须进入失败接收记录。

## 6. 配置文件需求

### 6.1 配置文件职责

配置文件只定义事件契约，不定义业务逻辑。

配置文件回答三个问题：

1. 谁可以提交事件。
2. 可以提交什么事件。
3. 事件 payload 应该长什么样，以及 Web 端如何展示。

### 6.2 来源系统配置

来源系统配置应支持：

1. `sourceSystem`：来源系统唯一标识。
2. `displayName`：来源系统展示名。
3. `enabled`：是否启用。
4. `allowedEventTypes`：允许提交的事件类型列表。

### 6.3 事件类型配置

事件类型配置应支持：

1. `eventTypeId`：事件类型 ID。
2. `displayName`：事件类型展示名。
3. `description`：事件类型描述。
4. `enabled`：是否启用。
5. `defaultSeverity`：默认严重级别。
6. `schemas`：schema 版本定义。

### 6.4 schema 字段配置

schema 字段配置应支持：

1. 字段名。
2. 字段 label。
3. 字段类型。
4. 是否必填。
5. 枚举值。
6. 最大长度。
7. 是否在列表展示。

V1 字段类型建议限制为：

1. `string`
2. `number`
3. `boolean`
4. `datetime`
5. `enum`
6. `object`
7. `array`

### 6.5 V1 不支持的配置能力

1. 鉴权密钥。
2. 动态规则表达式。
3. 条件必填。
4. 跨字段复杂校验。
5. 自动告警条件。
6. 工单映射。
7. 处置动作。
8. 外部系统回调。
9. 用户自助修改配置。
10. 热更新强一致承诺。

## 7. API 行为需求

### 7.1 提交事件

API 用于接收内部系统提交的安全事件。

请求体至少包含：

```json
{
  "sourceSystem": "iam-service",
  "eventId": "evt-20260612-0001",
  "eventTypeId": "LOGIN_BRUTE_FORCE",
  "schemaVersion": "1.0",
  "severity": "high",
  "occurredAt": "2026-06-12T10:00:00+08:00",
  "summary": "用户 admin 在 5 分钟内连续登录失败 20 次",
  "payload": {
    "username": "admin",
    "srcIp": "10.0.0.12",
    "failCount": 20
  }
}
```

### 7.2 成功响应

合法事件持久化成功后，API 返回成功。

响应至少表达：

1. 是否成功。
2. 事件实例 ID。
3. 是否为重复提交。

### 7.3 失败响应

请求失败时，API 必须返回明确失败原因。

失败原因至少覆盖：

1. 缺少基础必填字段。
2. 未知来源系统。
3. 来源系统已禁用。
4. 来源系统无权提交该事件类型。
5. 未知事件类型。
6. 未知 schema 版本。
7. payload 必填字段缺失。
8. payload 字段类型错误。
9. payload 枚举值非法。
10. 幂等冲突。
11. 数据库写入失败。

## 8. Web 页面需求

### 8.1 列表页定位

事件列表页面向安全运营人员，定位为安全事件收件箱。

打开列表后，安全运营人员必须能快速判断：

1. 发生了什么。
2. 来自哪里。
3. 严重程度如何。
4. 何时发生、何时被接收。
5. 是否需要进入详情继续查看。

### 8.2 列表默认行为

1. 默认按 `receivedAt` 倒序展示。
2. 默认不按严重级别过滤。
3. 不展示所有 payload 字段。
4. 只展示配置中显式标记为列表字段的 payload 字段。

### 8.3 列表字段

列表页必须展示：

1. 接收时间 `receivedAt`
2. 发生时间 `occurredAt`
3. 来源系统 `sourceSystem`
4. 事件类型展示名
5. 严重级别 `severity`
6. 摘要 `summary`
7. 事件实例 ID `eventId`

### 8.4 列表筛选

V1 必须支持：

1. 时间范围筛选。
2. 来源系统筛选。
3. 事件类型筛选。
4. 严重级别筛选。

V1 暂不支持：

1. 全文搜索。
2. 高级条件组合。
3. 保存筛选器。
4. 自定义列。
5. 导出。

### 8.5 详情页

详情页必须展示三类信息：

1. 基础事实。
2. 结构化 payload。
3. 原始 payload。

基础事实必须包含：

1. `eventId`
2. `sourceSystem`
3. `eventTypeId`
4. 事件类型展示名
5. `schemaVersion`
6. `severity`
7. `summary`
8. `occurredAt`
9. `receivedAt`

结构化 payload 展示规则：

1. 使用配置中的字段 label。
2. 不只显示原始字段名。
3. 配置外字段单独放在未定义字段区域。

原始 payload 展示规则：

1. 只读展示。
2. 大 payload 应折叠展示或限制展示长度，避免页面卡死。

### 8.6 详情页定位

1. 列表每条事件必须能进入详情页。
2. 详情页 URL 应能重新打开并定位到同一事件。
3. 页面展示上必须显示 `sourceSystem + eventId`。

## 9. 失败接收记录需求

非法事件不进入业务事件列表，但必须保留失败接收记录。

失败接收记录至少包含：

1. 接收时间。
2. 请求中的来源字段。
3. 请求中的事件类型字段。
4. 失败原因。
5. 请求摘要。

失败记录必须覆盖：

1. 来源不允许。
2. 事件类型不允许。
3. schema 版本不允许。
4. payload 校验失败。
5. 幂等冲突。
6. 系统内部错误。

对于超大非法 payload，系统必须避免完整无限制保存导致存储失控。

## 10. 验收标准

### 10.1 API 提交验收

| 用例 | 控制点 | 观测点 |
| --- | --- | --- |
| 合法事件提交成功 | 内部系统提交完整合法事件 | API 返回成功，事件被保存，Web 列表可见 |
| 缺少必填基础字段 | 缺少 `sourceSystem` 或 `eventTypeId` | API 返回失败，指出缺失字段，业务列表不可见 |
| 调用方不传 `eventId` | 合法事件缺少 `eventId` | 后台生成 `eventId`，事件保存成功 |
| 调用方提交 `receivedAt` | 请求体包含 `receivedAt` | 后台忽略该值，使用服务端接收时间 |

### 10.2 配置校验验收

| 用例 | 控制点 | 观测点 |
| --- | --- | --- |
| 来源系统在清单中 | 已配置 `sourceSystem` 提交事件 | 继续进入事件类型和 payload 校验 |
| 来源系统不在清单中 | 未知 `sourceSystem` 提交事件 | API 拒绝，失败记录可查 |
| 来源系统提交未授权事件类型 | 已知来源提交未授权 `eventTypeId` | API 拒绝，原因明确 |
| 未知事件类型 | 提交不存在的 `eventTypeId` | API 拒绝 |
| 未知 schema 版本 | 提交不存在的 `schemaVersion` | API 拒绝 |
| payload 缺少必填字段 | 缺少配置必填字段 | API 拒绝，返回缺失字段名 |
| payload 字段类型错误 | `number` 字段传入字符串 | API 拒绝，返回类型错误 |
| payload 枚举值非法 | enum 字段传入非法值 | API 拒绝，返回允许值范围 |
| payload 包含配置外字段 | payload 包含未定义字段 | 事件保存，详情页未定义字段区域展示 |

### 10.3 持久化验收

| 用例 | 控制点 | 观测点 |
| --- | --- | --- |
| 成功返回前完成持久化 | 提交合法事件后立即查询列表 | 事件可见 |
| 数据库写入失败 | 模拟数据库不可写 | API 返回失败，不能返回成功 |
| 重复提交完全相同事件 | 同一 `sourceSystem + eventId` 重复提交相同内容 | API 返回成功，业务列表只出现一条 |
| 重复 eventId 但 schema 不同 | 同一幂等键提交不同 `eventTypeId` 或 `schemaVersion` | 第二次请求被拒绝，记录冲突 |
| 重复 eventId 且字段值不同 | 同一幂等键提交不同 payload 字段值 | 第二次请求被拒绝，记录冲突 |
| 不同来源使用相同 eventId | 不同 `sourceSystem` 使用相同 `eventId` | 两条事件均保存成功 |

### 10.4 Web 列表验收

| 用例 | 控制点 | 观测点 |
| --- | --- | --- |
| 默认列表展示 | 安全运营打开事件列表页 | 按 `receivedAt` 倒序展示 |
| 列表展示核心字段 | 查看任意事件列表项 | 可见接收时间、发生时间、来源系统、事件类型、严重级别、摘要、事件 ID |
| 按来源系统筛选 | 选择某个 `sourceSystem` | 只展示该来源事件 |
| 按事件类型筛选 | 选择某个事件类型 | 只展示该类型事件 |
| 按严重级别筛选 | 筛选 `high` 或 `critical` | 只展示对应严重级别事件 |
| 按时间范围筛选 | 选择时间范围 | 只展示范围内事件 |

### 10.5 Web 详情验收

| 用例 | 控制点 | 观测点 |
| --- | --- | --- |
| 进入详情页 | 点击列表中的某条事件 | 进入对应事件详情页 |
| 详情 URL 可复用 | 复制详情 URL 后重新打开 | 仍能定位同一事件 |
| 展示基础事实 | 打开事件详情 | 可见基础事实字段 |
| 展示结构化 payload | 字段配置了 label | 使用 label 展示字段 |
| 展示未定义字段 | payload 包含配置外字段 | 未定义字段区域可见 |
| 展示原始 payload | 打开详情页查看原始内容 | 可见原始提交 payload |
| 大 payload 不拖垮页面 | 提交较大 payload | 页面可用，内容折叠或限制展示 |

### 10.6 失败记录验收

| 用例 | 控制点 | 观测点 |
| --- | --- | --- |
| 非法事件不进入业务列表 | 提交未知来源、未知事件类型或 schema 错误事件 | Web 业务事件列表不可见 |
| 失败记录可追溯 | 提交非法事件后查看失败记录 | 可见接收时间、来源字段、事件类型字段、失败原因和请求摘要 |
| 超大非法 payload | 提交超大非法 payload | 系统拒绝或安全记录摘要，不造成存储或页面不可用 |

## 11. 决策记录

1. V1 接受“内部系统 + 不鉴权”的前提，但必须通过 `sourceSystem` 配置清单约束来源。
2. `eventId` 定义为单次事件实例 ID，不允许同一幂等键对应不同事件事实。
3. payload 配置外字段允许保存，但必须标记为未定义字段。
4. `summary` 由调用方提交，后台不做模板生成。
5. 合法事件返回成功前必须完成持久化。
6. 非法事件不进入业务事件列表，但进入失败接收记录。
7. 暂不考虑配置变化后的历史 schema 展示策略。
8. 暂不设置默认保存期限。
